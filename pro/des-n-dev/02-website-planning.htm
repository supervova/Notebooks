Планирование веб-сайта

До начала работ нужно совместно с клиентом составить письменное техническое задание (request for proposal — RFP). Не 100-страничную технико-юридическую книгу, а краткий сборник задач с исходными данными, сделанный на основе прошлых ТЗ и шаблонов. На этапе стабильного развития для крупных проектов надо оформлять эту работу отдельным договором и брать за неё деньги.

Если ТЗ оплачивается отдельно, то пишется подробно, так, чтобы мог использовать сторонний разработчик. Если ТЗ не оплачивается, то составляется в краткой форме.

  1. Фундаментальные определения:
    • общие маркетинговые задачи и послания, место веб-сайта в маркетинговой программе;
    • потенциальные потребители;
    • конкурентная среда;
    • контекст использования — типичные устройства, внешняя среда;
    • интегрированная оценка существующего сайта (в малобюджетных проектах — краткий SWOT на основе этой оценки).
  2. Содержание и авторы (источники):
    • Тексты
    • Изображения (для образов можно использовать снимки из библиотек и Flickr; для фотографий продуктов, интерьеров надо нанимать профtccионалов; если бюджет сркомный, можно нанимать студентов фотостудий)
    • Аудиоматериалы
    • Видеоматериалы
    • Flash-материалы
    • Документы PDF и Word
    • Внешние (входщие) RSS-ленты
    • Twitter-ленты
    • Facebook-ленты.
  3. Если сайт большой со сложными функциями (веб-приложение), надо выяснить:
    • Какой хостинг нужен?
    • Как будет проходить процесс регистрации/авторизации?
    • Как будут осуществляться резервное копирование и хранение данных?
    • Нужно ли использовать уже существующие данные? Если да, то как?
  4. Общие рекомендации по одному или нескольким аспектам разработки и эксплуатации сайта:
    • Технологии
    • Информационная архитектура
    • Маркетинговые стратегии
    • Бизнес-процессы.
    Там, где решение очевидно, рекомендация должна быть одна. Там где возможны альтернативы, нужно их предлагать, перечисляя все плюсы и минусы.
  5. Предложения по реализации каждого типа материалов, каждого раздела. Структура каждой рекомендации должна включать MVP и «полную комплектацию».
  6. График проекта — в диаграмме Гантта (см. шаблон в Merlin) или электронной таблице Google. С учетом MVP и «полной комплектации».

SMART-цели для каждого типа страницы: первой, титульной страницы раздела, внутренней, страницы продукта и т.д

См. также «Создание персонажей целевых аудиторий»

Применительно к веб-страницам аббревиатура SMART немного видоизменяется:

  1. Specific — конкретная.
  2. Measurable — измеримая в конексте количества.
  3. Actionable — с очевидными решениями, которые проистекают из сути цели (если цель — увеличить кросс-продажи, очевидным решением будет предоставить продемонстрировать похожие продукты)
  4. Relevant — релевантными конкретной странице, разделу.
  5. Trackable — измеримая в контексте возможности подсчета.

Для каждой страницы составляются цели пользователей и цели бизнеса. Например.

Главная цель пользователя (ЦП1): я хочу побольше узнать о продукте, его дизайне, характеристиках, спецификациях — соотвествует ли он моим потребностям, предпочтениям и финансовым возможностям.

Вторая цель пользователя (ЦП1): я хочу купить этот товар.

Главная бизнес-цель (БЦ1): мы хотим увеличить количество добавлений продукта в «корзину» на 20%.

Вторая бизнес-цель (БЦ2): мы хотим увеличить количество кросс-продаж с этой страницы на 20%.

Для формулировки целей надо пообщаться с предполагаемыми пользователями и ответственными лицами. Затем записать цели в электронную таблицу, во втором столбце которой собирать возможные решения. Например, для ЦП1 это может быть:

  • Изображение продукта целиком
  • Увеличенные изображения деталей продукта
  • Описание продукта и его преимуществ
  • Спецификации, релевантные потребностям потребителей подобных продуктов
  • Модели и опции (цвет, размер)
  • Цена.

После этого, можно составлять каркасную модель, согласно иерархии целей (приоритеты на страницы отдаются главным целя пользователя и бизнеса).
